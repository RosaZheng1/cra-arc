Analyze data

In this phase, all available data is analyzed to identify issues with the content and navigation of the relevant web pages. The data that may be analyzed includes:

  • page content
  • User Feedback Tool data
  • research Data
  • web analytics
  • SEO data
  • existing UX testing results
  • Call Centre data
  • social media

User feedback data

The feedback tool is a mechanism embedded into specific web page on Canada.ca. Users can use the feedback tool to report issues with content on a page.

Specific factors to examine

  • Compare call drivers with user feedback data and note any inconsistencies or overlap
  • Verify the problem statements from program areas with the data from the feedback tool to check for issue alignment
  • Assess recurring issues that affect multiple users
  • Determine if there are content gaps (missing content)
  • Identify pages that may need to be linked or merged
  • Analyze findability and information architecture issues with existing content
  • Prioritize the issues found to create a preliminary usability test plan

Call Centre data

Phone calls received by the Call Centre are analyzed to determine the reason for the call. It’s important to know:

  • why the user is calling
  • what task they were trying to complete on Canada.ca and what barriers they encountered with the web pages/content that led to them to call

Call drivers

Knowing why individuals are contacting the Call Centre helps to determine areas where there are issues with the web content.

Call drivers with high traffic on their related web content could indicate that the:

  • web content is difficult to understand, so users call for clarification
  • web content is insufficient, so users have to call to complete their tasks
  • the user found the wrong web page and as a result, couldn’t find the information they require

Call drivers with low traffic on their related web content could indicate that users have difficulty searching and give up before finding what they need.

Call drivers that have little-to-no related web content may only be solvable by calling. In this case, there may be an opportunity to create new content to address users’ needs.

News articles and social media

News articles and social media platforms such as YouTube, Facebook, and Reddit are frequently monitored for mentions related to the relevant CRA web services to identify pages involved recurring issues users may be experiencing.

Host a task scenario workshop

The issues identified when analyzing the data from the various sources mentioned above are used to create job stories. These are developed during a task scenario workshop with input from subject matter experts (SMEs) and the project team.

Objective of the task scenario workshop

The objective of the workshop is to identify real world problems by crafting job stories which will help in creating tasks that users will attempt to complete during user testing.

Job stories

Job stories are used to craft the task scenarios.

A job story is a tool that helps you understand the user’s needs when they interact with the website.

In general, a job story is structured based on the idea that an individual will use a product in a particular situation, to achieve a desired outcome:

When <situation>, I want to <motivation>, so I can <expected outcome>

Example: Job story

When I’m feeling ill, I want to check if it is a cold or flu, so I can decide whether to stay home.

To create a job story, the situation, motivation, and outcome, is informed by the following:

  • Call Centre data and trends:
    • Why are users calling?
    • What were they trying to do before deciding to call?
  • web analytics:
    • What information are users seeking on the website and why didn’t they find it?
    • How are they navigating through our current content?
  • subject matter experts:
    • What do our users need to do to?
    • How do they currently perform the task?
    • Where do they struggle in the process?

Creating task scenarios

Once job stories have been crafted, the team will create task scenarios based on the highest priority that's required for testing.

What task scenarios are

During user testing, participants are asked to complete tasks. A task is an activity a user would typically perform when using the website. For example, searching Canada.ca, to learn about deadlines and installment payments.

A scenario provides an explanation and context as to why the test participant is completing the task.

Combining the action the participant will attempt and the context for completing it, creates a task scenario.

Example: Task scenario

Sandy isn’t sure how to file a tax return since it’s her first time not using an accountant. Her friend told her to use TurboTax. Can Sandy file her 2020 taxes online with TurboTax?

The task scenarios must:

  • be realistic: involve real tasks that users would typically perform on pages
  • be actionable: include a goal that user is working towards by completing the task
  • avoid giving away the answer: the task needs to be worded differently than the content on a page, so users don’t just match headings and words without reading.

Create a test plan for baseline usability testing

Once the task scenarios have been developed, the next step is to create a test plan. A test plan is a document that provides details about the testing, as well as the post-test feedback that will be solicited. In the plan you will find:

  • when the test will be conducted
  • the program that will be used to administer the test (example: TryMyUI)
  • the type of testing – eg. moderated or unmoderated
  • which platform (mobile,desktop or both) will be tested
  • length of time participants have to complete the test
  • a list of task scenarios each participant will be asked to complete
  • the testing metrics (KPIs) that will be used:
    • a 20% point increase in both findability and task success, or a rate of 80% in findability and task success
    • Ease of use scale
    • System Usability Scale (SUS)
    • Time on task
  • Participants:
    • number of test participants; typically 18 to 20 individuals
    • user recruitment methods
    • recruitment screening criteria (for example, employed and unemployed individuals aged 18 to 65)

Conduct baseline usability testing

During baseline testing, users attempt to complete the task scenarios.

Baseline testing is conducted to review existing content and navigation of web pages in their current state. Seeing how the user interacts with the pages and the issues they encounter when attempting to complete the task scenarios helps to determine specific improvements that need to be made.

After testing is complete

Gather data in the UX scorecard template/Usability Performance Dashboard (UPD) to share with the stakeholders, including:

  • time on task
  • task success
  • ease of Use
  • SUS and Summary questionnaire & data
  • observations
  • Design, IA, content recommendations

In the next phase of the project (Designing), a prototype of the pages is created with the specific improvements identified in the baseline testing.
